## 二、核心改进点

### 2.1 "示例即约束"原则（新增）

#### 理论基础（来自 IDEA-006 修订）

```markdown
**原理说明**：
对于复杂的结构化输出（如 JSON Schema、格式化报告），**具体示例**比抽象描述对 LLM 的约束力强 10 倍。

**核心机制**：
- LLM 通过**模式匹配**学习任务要求
- 具体示例提供了**强锚点**（Anchoring Effect）
- 抽象描述容易被 LLM "自由发挥"
- 完整示例确保输出格式的**一致性和稳定性**

**验证效果**：
- 有完整示例：格式一致性 > 95%，工具调用合规率 > 95%
- 仅抽象描述：格式一致性 < 60%，工具调用合规率 < 70%
```

#### 实施内容（第 265-532 行）

新增 **3 个完整 JSON 示例**，每个示例包含：

1. **示例 1：SUCCESS 输出**（第 269-384 行）
   - 完整的工作流程：读取文件 → 搜索调用 → 执行测试 → 修复问题
   - 所有字段都填充了真实数据
   - evidence_summary 记录 3 次工具调用
   - claims ↔ evidence 完整关联
   - 包含人类可读摘要部分

2. **示例 2：BLOCKED 输出**（第 388-441 行）
   - 无证据时的标准输出
   - tool_calls_this_turn = 0
   - blocked_reason 详细说明缺失的验证
   - required_tool_calls 列出需要的工具

3. **示例 3：PARTIAL 输出**（第 444-532 行）
   - 工作中途状态
   - 包含测试失败的真实场景
   - next_action 指示继续修复

**效果**：
- 从"抽象规范"变成"具体模板"
- LLM 可以直接模仿示例结构
- 大幅降低格式偏差风险

### 2.2 工具纪律强化（Layer 1，第一轮改进）

#### 改进前（第 25 行）
```markdown
- 工具纪律：**先查证后输出；先调用再回答**。能用工具确认的内容先查证，
  输出必须带证据（文件路径/命令/关键日志行）；长上下文中间产物写入
  `.autoworkflow/state.md` 或 `.autoworkflow/tmp/feature-shipper-notes.md`，
  对话只保留摘要与引用。
```

**问题**：
- 只是"建议"，没有强制机制
- 没有具体的检查步骤
- 没有违反后的处理方式

#### 改进后（第 219-299 行）

新增内容：

1. **铁律表格**（第 225-231 行）
   - 明确 5 种陈述类型及对应工具
   - 用 ❌ 和 ✅ 对比错误/正确做法

2. **输出前自检**（第 233-242 行）
   - 4 个强制检查项
   - 每次输出前必须执行

3. **BLOCKED 输出格式**（第 244-265 行）
   - 检查失败时的标准输出
   - 包含 failed_checks、claims_without_evidence、required_tool_calls

4. **禁止的输出模式**（第 267-299 行）
   - 5 种具体的禁止模式
   - 每种都有错误示例 + 正确做法

**效果**：
- 从"建议"变成"强制"
- 有明确的检查步骤和阻断机制
- LLM 有清晰的行为边界

### 2.3 输出格式结构化（Layer 2，第一轮改进）

#### 改进前（第 219-235 行）
```markdown
## 输出格式（默认）

​```markdown
## 目标与验收标准
- ...

## 实施计划（3–7 项）
1. ...

## 进展与验证
- [x] 任务1：...（验证：...）
- [ ] 任务2：...

## 交付说明
- 验证命令：...
- 变更摘要：...
​```
```

**问题**：
- 纯 Markdown 自由文本
- 无法机器解析
- "有没有证据"不可检查

#### 改进后（第 303-399 行）

新增内容：

1. **结构化 JSON 输出**（第 307-366 行）
   - evidence_summary：记录本轮工具调用
   - claims：事实陈述列表
   - evidence：证据详情（与 claims 关联）
   - result：Agent 特定结果
   - next_action：下一步行动

2. **人类可读摘要**（第 368-384 行）
   - 保留原有的 Markdown 格式
   - 作为第二部分补充

3. **强制规则**（第 386-399 行）
   - tool_calls_this_turn = 0 → status 必须 BLOCKED
   - confidence = HIGH → 必须有 evidence_id
   - 违反格式 → 输出 BLOCKED

**效果**：
- 机器可解析 + 人类可读
- evidence_summary.tool_calls_this_turn 可验证"有没有调用工具"
- claims ↔ evidence 关联使证据可追溯

## 三、示例对比

  }

  "agent": "feature-shipper",

## 四、预期效果

### 4.1 第一轮改进效果（Layer 1 + Layer 2）

| 指标 | 改进前 | 第一轮后（预期） |
|------|-------|----------------|
| 幻觉输出率 | 高（无约束） | 中低（50-70% 减少） |
| 证据可追溯性 | 低（自由文本） | 高（claims ↔ evidence） |
| 输出可解析性 | 低（Markdown） | 高（JSON Schema） |
| 违规检测 | 不可能 | 可能（检查 tool_calls_this_turn） |

### 4.2 第二轮改进效果（+ "示例即约束"）

| 指标 | 第一轮后 | 最终版本（预期） | 提升 |
|------|---------|----------------|------|
| 格式一致性 | 中（<60%） | 高（>95%） | +35% |
| 工具调用合规率 | 中（<70%） | 高（>95%） | +25% |
| LLM 理解清晰度 | 抽象规范 | 具体模板 | 质变 |
| 输出偏差风险 | 中高 | 低 | 显著降低 |

### 4.3 总体效果评估

| 维度 | 效果 |
|------|------|
| **约束强度** | 弱（建议）→ 强（强制 + 示例锚定） |
| **幻觉阻断率** | 预期达到 70-80%（Layer 1+2+示例） |
| **格式稳定性** | 预期 >95%（示例锚定效应） |
| **实施成本** | 中（仅需更新 Prompt，无需代码） |
| **维护成本** | 低（示例自解释） |
