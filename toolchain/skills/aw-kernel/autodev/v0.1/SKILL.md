# AutoDev v0.1 基础版

> **版本目标**：验证核心工作流是否可行，建立用户信心
> **状态管理**：TodoWrite（内存）
> **回路层级**：Level 0 only

---

## 核心原则

1. **人类在环**：每个 Phase 结束请求用户确认
2. **状态可见**：使用 TodoWrite 追踪任务进度
3. **小步迭代**：每次只做小步修改，便于理解
4. **测试验证**：每个任务完成后必须测试
5. **失败透明**：Level 0 失败 3 次后立即请求人工介入

---

## 启动协议

1. **识别用户需求**
2. **输出欢迎信息**：
   ```
   🚀 AutoDev v0.1 已启动

   工作流：需求精炼 → 任务规划 → 迭代开发 → 验收提交
   回路支持：Level 0 即时修复（最多 3 次）

   开始 Phase 1: 需求理解与精炼...
   ```
3. **进入 Phase 1**

## Phase 1: 需求理解与精炼

### 目标
将模糊需求转化为清晰的验收标准（DoD）。

### 执行步骤

1. **分析用户输入**
   - 识别需求类型（新功能 / Bug 修复 / 重构 / 优化）
   - 提取核心价值命题

2. **使用 AskUserQuestion 澄清**（最多 3 轮）
   - 只问阻塞实现的关键问题
   - 避免过早讨论技术细节

   示例问题：
   - 核心价值是什么？解决什么问题？
   - 哪些是 must-have？哪些是 nice-to-have？
   - 验收标准是什么？怎么算完成？

3. **输出需求摘要**
   ```markdown
   ## 需求摘要

   **核心价值**：[一句话描述]

   **功能范围**：
   - ✅ 做什么：[列表]
   - ❌ 不做什么：[列表]

   **验收标准**（DoD）：
   - [ ] 标准1
   - [ ] 标准2
   - [ ] 标准3
   ```

4. **执行 G1 门禁检查**
   - 验收标准已明确（≥3 条，可测试）？
   - 功能范围已界定（有"不做什么"）？
   - 核心价值已清晰（能一句话解释）？
   - 用户已确认需求理解？

   **全部 ✅** → Phase 2
   **任一 ❌** → 补充信息，重新输出摘要

5. **请求用户确认**
   ```
   ✅ G1 门禁通过

   请确认以上需求理解是否正确？
   [确认 / 调整 / 重新说明]
   ```

## Phase 2: 任务规划

### 目标
将需求拆分为 3-7 个可执行的原子任务。

### 执行步骤

1. **分析项目结构**
   ```bash
   # 了解项目目录结构
   Glob("**/package.json")
   Glob("src/**/*")

   # 读取关键配置文件
   Read("package.json")
   Read("README.md")  # 如存在
   ```

2. **识别技术栈**
   - 语言/框架（React / Vue / Node / Python 等）
   - 测试工具（Jest / Mocha / Pytest 等）
   - 构建工具（Webpack / Vite / etc）

3. **拆分任务**（3-7 个）

   **任务拆分原则**：
   - 每个任务 ≤ 半天工作量
   - 任务之间有清晰的依赖关系
   - 每个任务有独立的完成标准
   - 任务边界清晰，形成可读 diff

4. **创建 TodoWrite 任务列表**
   ```javascript
   TodoWrite([
     {
       content: "任务1: [具体描述]",
       status: "pending",
       activeForm: "[进行中描述]"
     },
     {
       content: "任务2: [具体描述]",
       status: "pending",
       activeForm: "[进行中描述]"
     },
     // ... 3-7 个任务
   ])
   ```

5. **输出任务规划**
   ```markdown
   ## 任务规划

   | 任务ID | 任务描述 | 涉及文件 | 验证方式 |
   |--------|---------|---------|---------|
   | 1 | ... | ... | ... |
   | 2 | ... | ... | ... |
   | 3 | ... | ... | ... |

   **测试命令**：`npm test`（或项目对应命令）
   ```

6. **执行 G2 门禁检查**
   - 任务数量在 3-7 个之间？
   - 每个任务有完成标准？
   - 验证命令已确定？
   - 用户已确认？

   **全部 ✅** → Phase 3
   **任一 ❌** → 调整规划

7. **请求用户确认**
   ```
   ✅ G2 门禁通过

   请确认任务规划是否合理？
   [确认开始实现 / 调整规划]
   ```

## Phase 3: 迭代开发

### 目标
逐个完成任务，每个任务都经过测试验证。

### 执行步骤（对每个任务循环）

#### 3.1 开始任务

1. **更新任务状态为 in_progress**
   ```javascript
   TodoWrite([
     { content: "任务1", status: "completed", ... },
     { content: "任务2", status: "in_progress", ... },  // 当前
     { content: "任务3", status: "pending", ... },
   ])
   ```

2. **输出任务开始提示**
   ```
   📝 开始任务2: [任务描述]

   涉及文件：[列表]
   验证方式：[说明]
   ```

#### 3.2 执行代码修改

1. **理解现有代码**
   ```bash
   # 读取相关文件
   Read("src/components/LoginForm.tsx")

   # 搜索关键代码
   Grep("pattern", path="src/", output_mode="content")
   ```

2. **执行修改**
   - 使用 Edit 修改现有文件
   - 使用 Write 创建新文件
   - **保持小步修改**（每次 ≤50 行）

3. **说明修改内容**
   ```
   ✏️ 修改内容：
   - 文件1: [修改说明]
   - 文件2: [修改说明]
   ```

#### 3.3 执行 G3 门禁检查（带 Level 0 回路）

**初始化 retry 计数**：retry = 0

**测试循环**：

```
┌─────────────────────────────────────┐
│ 1. 执行测试                         │
│    Bash("npm test")                 │
│                                     │
│ 2. 分析结果                         │
│    ├─ ✅ 通过 → 标记任务 completed  │
│    │           进入下一任务         │
│    │                                │
│    └─ ❌ 失败 → retry++             │
│              ↓                      │
│         retry ≤ 3?                  │
│         ├─ Yes → Level 0 即时修复   │
│         └─ No  → 人工介入           │
└─────────────────────────────────────┘
```

**Level 0 即时修复**（retry ≤ 3）：

1. **分析错误信息**
   - 读取完整错误堆栈
   - 定位失败的具体测试

2. **定位问题原因**
   ```bash
   # 重新读取相关代码
   Read("src/file-with-error.ts")
   ```
   - 理解预期行为 vs 实际行为
   - 找到根本原因

3. **修复代码**
   - 使用 Edit 修改
   - 保持小步修改（≤10 行）
   - 说明修复思路

4. **重新测试**
   - retry++
   - 回到"执行测试"步骤

**人工介入**（retry > 3）：

输出调试受阻报告：

```markdown
## 🚨 调试受阻（Level 0 失败）

**任务**：[任务描述]

**问题**：[错误描述]

**已尝试**：
1. 方法1 - 结果：[失败原因]
2. 方法2 - 结果：[失败原因]
3. 方法3 - 结果：[失败原因]

**需要主人帮助**：
🔸 提供更多上下文或领域知识
🔸 调整任务目标或验收标准
🔸 跳过此任务，先处理其他任务
```

**等待用户指导**

#### 3.4 任务完成

任务测试通过后：

1. **标记任务 completed**
   ```javascript
   TodoWrite([
     { content: "任务1", status: "completed", ... },
     { content: "任务2", status: "completed", ... },  // 刚完成
     { content: "任务3", status: "in_progress", ... },
   ])
   ```

2. **输出完成提示**
   ```
   ✅ 任务2 已完成

   已完成：2/5 任务
   下一步：任务3
   ```

3. **进入下一任务**（如有）

#### 3.5 Phase 3 完成

所有任务完成后：

```
🎉 所有任务已完成！

完成情况：
- ✅ 任务1: [描述]
- ✅ 任务2: [描述]
- ✅ 任务3: [描述]
- ✅ 任务4: [描述]

进入 Phase 4: 验收提交
```

## Phase 4: 验收提交

### 目标
确保所有验收标准已满足，生成交付报告。

### 执行步骤

1. **执行 G4 门禁检查**

   对照 Phase 1 的验收标准逐项检查：

   ```markdown
   ## 验收检查

   - [ ] 验收标准1 - 验证方式：[说明]
   - [ ] 验收标准2 - 验证方式：[说明]
   - [ ] 验收标准3 - 验证方式：[说明]
   ```

2. **执行完整测试**
   ```bash
   # 运行所有测试
   Bash("npm test")

   # 如有 build 命令
   Bash("npm run build")
   ```

3. **识别遗漏项**

   如有验收标准未满足：
   - 列出遗漏项
   - 回到 Phase 3 补充实现
   - 或标记为"已知限制"并请求用户确认

4. **生成交付报告**
   ```markdown
   ## 🎯 交付报告

   **实现内容**：
   - [x] 验收标准1
   - [x] 验收标准2
   - [x] 验收标准3

   **变更文件**：
   - src/file1.ts (+50, -10)
   - src/file2.tsx (+30, -5)

   **测试结果**：
   ✅ 所有测试通过

   **验证方式**：
   ```bash
   npm test
   ```

   **已知限制**：
   - [如有]
   ```

5. **请求 Git 提交确认**

   ```
   ✅ 所有验收标准已满足

   是否创建 Git commit？
   [是 / 否 / 先让我检查]
   ```

   **如用户确认 "是"**：
   - 使用 `/git-commit` Skill 创建 Conventional Commit
   - 输出 commit 信息

6. **输出完成信息**
   ```
   🎉 AutoDev v0.1 工作流完成！

   感谢主人的信任喵～ ฅ'ω'ฅ
   ```

## 工具纪律（IDEA-006）

遵循 "No Evidence, No Output" 原则：

| 陈述类型 | 必须的工具调用 |
|---------|--------------|
| "文件 X 存在/不存在" | Glob 或 Read |
| "代码中有/没有 Y" | Grep 或 Read |
| "函数 Z 的逻辑是..." | Read（具体行号） |
| "测试通过/失败" | Bash（实际运行） |

**输出前自检**：
- □ 每个事实陈述都有本轮工具证据吗？
- □ 有没有假设文件存在而没有验证？
- □ 有没有推断代码行为而没有读取？

**任一检查失败 → 先执行工具调用，再继续**

---

## 禁止行为

- ❌ 凭空陈述文件内容（必须先 Read）
- ❌ 假设性搜索（必须先 Grep/Glob）
- ❌ 跳过验收标准确认
- ❌ 跳过用户确认直接提交代码
- ❌ Level 0 失败后继续硬撑（必须在 retry > 3 时请求人工介入）

---

## v0.1 局限性说明

本版本**不支持**以下功能（将在后续版本实现）：

- ❌ Git 检查点回退（Level 1）
- ❌ 任务重构回路（Level 1）
- ❌ 整体重规划回路（Level 2）
- ❌ 状态持久化（.autodev/state.json）
- ❌ 指令模块化（phases/ gates/ loops/）

**如遇到 Level 0 无法解决的问题，请立即请求人工介入。**
