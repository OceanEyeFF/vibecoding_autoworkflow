# AUTODEV 小需求更稳工作流

> **版本**：v1.0
> **最后更新**：2026-01-13
> **维护者**：浮浮酱
> **配套文档**：[实施参考](AUTODEV_实施参考.md) | [分析精华](docs/analysis/autodev-insights.md)

---

## 0. 快速理解（10秒版）

**一句话总结**：入口规模 Gate + 需求2/实现3/交付1 + 证据型交付 = 小需求更稳

**适用范围**：单个小需求/小任务（建议 0.5～2 小时可闭环交付）

**核心理念**：
- 大任务挡在入口处（规模过大→拒绝/拆分）
- 不确定性前移量化（需求段固化为契约）
- 从"口头通过"到"证据通过"（实现段产出最小证据集）
- 职责单一（避免自证闭环）

**非目标**：不追求覆盖大项目（大任务先拆分为多个小需求）

---

## 1. 为什么需要"小需求更稳"

### 1.1 观察到的问题

- 小需求本应更容易做好：上下文少、边界清晰、验证成本低
- 当前主要问题：**缺少可验证契约、强制门禁、证据交付、修复闭环**
- 问题后置：缺陷在后段才暴露，返工成本被放大

> 详细失败模式分析：见 [docs/analysis/autodev-insights.md](docs/analysis/autodev-insights.md)

### 1.2 设计目标（面向"稳"）

| 目标 | 手段 | 效果 |
|------|------|------|
| **把大任务挡在入口** | 规模 Gate（SizeScore） | 过大→拒绝/拆分，不进默认路径 |
| **把不确定性前移** | 需求契约（约束/验收/验证方法） | 实现前固化可验证标准 |
| **从口头到证据** | 最小证据集（三件套） | 可复现、可审计 |
| **职责单一** | 2-3-1 角色编排 | 避免"自产自证自宣称" |

### 1.3 非目标（避免过度设计）

- ❌ 不追求覆盖所有大项目形态（大任务→拆分后多轮小需求）
- ❌ 不追求增加无限 Agent（以最少角色达到可审计与可复现）
- ❌ 不追求多里程碑规划（规模过大由入口 Gate 拆分解决）

## 2. 工作流结构

### 2.1 总体流程

```
入口规模 Gate（G0）
  ↓（放行/拆分/拒绝）
需求段（2 Agent）
  ├─ Spec-Owner：产出需求契约
  └─ Spec-Gatekeeper：审查放行（G1）
  ↓
实现段（3 Agent + 可选回路）
  ├─ Implementer：实现（不自证）
  ├─ Reviewer：静态审查（挑错）
  └─ Verifier：证据验证（G3）
  ↓
交付段（1 Agent）
  └─ Finalizer：汇总交付（G4）
```

### 2.2 入口规模 Gate（G0）

**目的**：防止大任务硬塞进小需求工作流，导致"看起来做了，实际没闭环"

**SizeScore 算法（0-10）**：

命中一项 +1 分：

1. 多模块/多子系统（不止一个清晰边界）
2. 多语言/多运行时（如 C++ + Python + Unity）
3. 并发/内存安全/无锁算法（天然高风险）
4. 性能硬指标（吞吐/延迟/GPU 时间等必须对标）
5. 跨平台矩阵 > 1（至少两种 OS/架构/设备）
6. 引入新依赖/新框架/新基础设施
7. 验收标准不清或不可验证
8. 预期改动文件数 > 5
9. 预期新增/修改代码量 > 300 LOC
10. 需要外部系统联调/长链路数据（无法本轮闭环）

**决策阈值**：

| SizeScore | 决策 | 输出 |
|-----------|------|------|
| **≤ 3** | ✅ 放行 | 进入需求段 |
| **4-6** | ⚠️ 必须拆分 | 拆分建议（≤5 个子任务） |
| **≥ 7** | ❌ 建议拒绝 | 明确拒绝理由 + 拆分/降低风险路径 |

**拆分模板**：

每个子任务包含：
- 目标与非目标
- 验收标准（可验证）
- 预计改动范围（文件/模块）
- 风险点与验证方式

### 2.3 门禁体系（G0-G4）

| 门禁 | 位置 | 检查内容 | 不通过处理 |
|------|------|---------|-----------|
| **G0** | 入口 | 规模/风险可控 | 拒绝/拆分 |
| **G1** | 需求段后 | 术语澄清、MUST-FOLLOW/MUST-VERIFY 齐全、平台矩阵明确 | 补充/回退 |
| **G3** | 实现段后 | 证据完整、约束逐项对照、P0 无遗漏 | 修复回路（最多3次） |
| **G4** | 交付段后 | 交付物清单、证据索引、风险/回退明确 | 补齐/标记未达标 |

### 2.4 最小证据集（Small 档位）

**必须产出（三件套）**：

1. **`constraint-compliance.md`**：约束符合度
   - 对 MUST-FOLLOW / MUST-VERIFY 逐项标记：满足/不满足/无法验证
   - 每项附证据链接或说明

2. **`verification-report.md`**：验证报告
   - 编译验证：通过/失败/无法验证 + 命令/日志摘要/原因
   - 测试验证：通过/失败/无法验证 + 命令/日志摘要/原因
   - 数值验证：通过/失败/无法验证 + 计算/脚本/容差
   - 性能验证（若有 MUST-MEASURE）：基准数据与对标

3. **`risk-matrix.md`**：验证矩阵与风险
   - 平台/环境矩阵：已验证/未验证/仅推断
   - 未验证项风险等级与建议验证路径

**可选（性能任务）**：
- `performance-baseline.md`：性能基准与对标数据

## 3. 角色定义

### 3.1 全局约定

#### 3.1.1 产物命名标准

为便于 Hooks/Skills 自动检查，统一采用固定文件名：

| 产物 | 产出者 | 用途 |
|------|-------|------|
| `requirement-contract.md` | Spec-Owner | 需求契约 |
| `spec-gate-decision.md` | Spec-Gatekeeper | 审查决策 |
| `implementation-notes.md` | Implementer | 实现说明（可选） |
| `review-findings.md` | Reviewer | 审查问题清单 |
| `constraint-compliance.md` | Verifier | 约束符合度 |
| `verification-report.md` | Verifier | 验证报告 |
| `risk-matrix.md` | Verifier | 风险矩阵 |
| `delivery-summary.md` | Finalizer | 交付摘要 |

#### 3.1.2 通用硬规则

所有 Agent 必须遵守：

1. **证据优先**：能验证的必须验证；不能验证必须写清原因、影响与风险等级
2. **禁止过度承诺**：无证据不得宣称"编译成功/测试通过/性能达标"
3. **结构化输出**：每次输出至少包含：
   - 产物清单（写了哪些文件/做了哪些检查）
   - 验证状态（通过/失败/无法验证 + 原因）
   - 风险与下一步（如何补证据/如何复现/如何回退）
4. **约束标签化**（写进 `requirement-contract.md`）：
   - `[MUST-FOLLOW]`：不可简化的设计/接口/行为约束
   - `[MUST-VERIFY]`：必须验证的正确性约束（含方法/容差/场景）
   - `[MUST-MEASURE]`：必须度量的性能/资源指标（含基准方法/环境）

#### 3.1.3 禁止自证闭环

- 避免"同一 Agent 既产出又审查又宣称通过"
- 放行必须由 Gatekeeper/Verifier 的证据判定
- Implementer 不宣称"已通过"
- Reviewer 不直接修改实现
- Verifier 不修代码（只产证据）

---

### 3.2 Intake-Gatekeeper（入口规模 Gate）

**定位**：autodev 入口处的规模/风险把关者

**输入**：
- 用户任务描述
- 可选：目标仓库/目录、已有上下文摘要

**输出**：
- `intake-assessment.md`
  - `SizeScore(0-10)` 评分明细（命中项列表）
  - 结论：允许进入 / 必须拆分 / 建议拒绝
  - 拆分建议（≤5 个子任务）与每个子任务的验收/风险/验证方式

**放行标准**：
- `SizeScore <= 3`：放行进入 Spec-Owner
- `SizeScore ∈ [4, 6]`：不放行，输出拆分建议
- `SizeScore >= 7`：建议拒绝执行

**边界（不做什么）**：
- 不写实现代码
- 不定义详细技术方案（把方案留给需求段）

---

### 3.3 Spec-Owner（需求契约产出者）

**定位**：把"自然语言需求"转成可执行、可验证、可传递的"需求契约"

**输入**：
- 用户需求
- Intake-Gatekeeper 的评估结果（若有）
- 现有代码/环境信息（若可用）

**输出**：
- `requirement-contract.md`（必须包含）：
  - `[MUST-FOLLOW]`：关键不可简化点（类型/接口/行为/架构边界）
  - `[MUST-VERIFY]`：必须验证的正确性项（含验证方法/容差/场景）
  - `[MUST-MEASURE]`：必须度量项（若存在性能硬指标）
  - 环境与平台矩阵：目标平台、工具链假设、验证可行性
  - 验收标准：可判定、可复现
  - 风险与降级策略：当验证不可做时的"最低证据替代方案"

**放行标准**：
- 每条 MUST-VERIFY 都包含：验证方法 + 容差/阈值 + 测试场景
- 所有关键术语无歧义（必要时给问卷式澄清选项）

**边界（不做什么）**：
- 不写实现代码
- 不替下游做"验证结果宣称"

---

### 3.4 Spec-Gatekeeper（需求契约审查者）

**定位**：对 `requirement-contract.md` 做可验证性与一致性审查

**输入**：
- `requirement-contract.md`
- 可选：用户原始需求（用于对照）

**输出**：
- `spec-gate-decision.md`
  - 结论：通过 / 需补充 / 需拆分 / 建议拒绝
  - 问题清单：按严重度（P0/P1/P2）
  - 必须补充的问题（缺一不可）

**放行标准**：
- 术语无歧义（或已明确采用某个可验证定义）
- MUST-VERIFY 具备验证方法与容差
- MUST-FOLLOW 足够清晰，能约束后续"合理化简"
- 约束之间不冲突
- 在小需求时间窗内可闭环（否则触发拆分）

**边界（不做什么）**：
- 不重写 Spec-Owner 的契约（只给审查意见与补充点）
- 不写实现代码

---

### 3.5 Implementer（实现者）

**定位**：严格对照需求契约实现交付物。只负责"做出来"，不负责"自证通过"

**输入**：
- `requirement-contract.md`（必须完整注入/可见）
- `spec-gate-decision.md`（若包含必须遵守的补充点）

**输出**：
- 实现产物（代码/配置/必要文档）
- `implementation-notes.md`（建议，但不强制）
  - 关键实现决策点
  - 与 MUST-FOLLOW 的对照说明
  - 已知限制（尤其是环境限制）

**放行标准**：
- 不允许绕开 MUST-FOLLOW
- 不允许省略 MUST-VERIFY 对应的"可验证接口/脚手架"（例如测试入口、可运行命令、基准脚本）

**边界（不做什么）**：
- 不宣称"测试通过/性能达标/跨平台已验证"
- 不把验证责任转嫁给 Reviewer（Reviewer 只做静态审查）

---

### 3.6 Reviewer（静态审查者）

**定位**：对照需求契约做静态审查（逻辑/边界/一致性/可维护性）

**输入**：
- 实现产物
- `requirement-contract.md`
- 可选：`implementation-notes.md`

**输出**：
- `review-findings.md`（必须）
  - **P0**：违反 MUST-FOLLOW、明显逻辑错误、安全风险、不可验证风险
  - **P1**：边界条件遗漏、潜在缺陷、质量门禁缺失
  - **P2**：可维护性/一致性改进建议
  - 每条必须含：定位、原因、建议修复、建议验证方式

**放行标准**：
- Reviewer 自身不"放行"，但必须明确标记是否存在 P0

**边界（不做什么）**：
- 不直接修改实现（除非明确允许 Reviewer 产出补丁）
- 不进行运行时验证（交给 Verifier）

---

### 3.7 Verifier（证据验证者）

**定位**：用证据回答"到底过没过"。把争议变成可判定事实

**输入**：
- 实现产物
- `requirement-contract.md`
- 可选：`review-findings.md`（用来优先验证 P0 风险点）

**输出（最小证据集）**：
- `verification-report.md`（必须）
- `constraint-compliance.md`（必须）
- `risk-matrix.md`（必须）
- `performance-baseline.md`（可选，若存在性能指标则建议必出）

**放行标准**：
- 存在 P0 不通过项 → 必须判定为"不放行"
- 若无法验证关键 MUST-VERIFY → 必须标记为"不具备通过证据"

**边界（不做什么）**：
- 不修代码（修复由 Implementer 承担；Verifier 只产证据与判定）

---

### 3.8 Finalizer（交付整合者）

**定位**：把散落的产物整合成"可交付、可复审、可回退"的最终包

**输入**：
- `requirement-contract.md`
- 实现产物
- `review-findings.md`
- `verification-report.md`
- `constraint-compliance.md`
- `risk-matrix.md`
- 可选：`performance-baseline.md`

**输出**：
- `delivery-summary.md`（必须）
  - 交付物清单（文件/入口/使用方式）
  - 验证摘要（通过/失败/无法验证）与证据索引
  - 未满足项与风险（按 P0/P1）
  - 回退/降级方案（怎么回到安全状态）
  - 下一步建议（例如需要的真实设备验证、需要补齐的环境）

**放行标准**：
- 如果存在 Verifier 判定"不放行"的 P0：Finalizer 必须阻止"完成宣称"

**边界（不做什么）**：
- 不引入新需求
- 不改实现（只整合与归档）

## 4. 运行策略

### 4.1 默认档位：2-3-1（小需求更稳）

**适用**：常规小需求（业务逻辑、小工具脚本、小范围改动、可本地验证）

**编排**：
- 需求段：2 Agent（Spec-Owner → Spec-Gatekeeper）
- 实现段：3 Agent（Implementer → Reviewer → Verifier）
- 交付段：1 Agent（Finalizer）

### 4.2 降级档位：2-2-1（验证不可做时）

**触发条件**：验证环境缺失且短期无法补齐

**编排**：
- 需求段：2 Agent（同默认）
- 实现段：2 Agent（Implementer → Reviewer，Verifier 缺失）
- 交付段：1 Agent（Finalizer）

**约束**：
- 必须输出"验证缺失清单 + 风险标记"
- Finalizer 必须显式提示"不可承诺通过"

### 4.3 升级档位：2-3-2（优化段加 Gatekeeper）

**触发条件**：优化方案多、互相冲突、存在较高集成风险

**编排**：
- 需求段：2 Agent（同默认）
- 实现段：3 Agent（同默认）
- 交付段：2 Agent（Finalizer + Integration-Gatekeeper）

**Integration-Gatekeeper 职责**：
- 方案优先级排序（价值/风险/依赖）
- 集成步骤（可操作）
- 验证计划（每步怎么验证）
- 回退方案（失败如何回滚）

### 4.4 Level 0 回路（修复闭环）

**触发条件**：
- G3 不通过
- Reviewer 给出 P0
- Verifier 验证失败

**行为**：
1. 生成"修复任务单"
2. Implementer 修复
3. Reviewer/Verifier 复检

**终止条件**：
- P0 清零且 Gate 通过 → 进入下一阶段
- 连续失败 3 次 → 请求人工介入（明确卡点与下一步建议）

## 5. 成功度量

### 5.1 核心指标

| 指标 | 目标值 | 用途 |
|------|-------|------|
| **问题发现时机** | ≥80% 在实现段暴露 | 避免问题后置 |
| **证据覆盖率** | 100% 产出三件套 | 可审计性 |
| **诚实度** | 0 次无证据宣称 | 避免虚假通过 |
| **返工成本** | 下降 ≥30% | 提升效率 |

### 5.2 门禁有效性

| 门禁 | 检查内容 | 阻断率目标 |
|------|---------|----------|
| **G0** | 规模过大 | >20% 大任务被拒绝/拆分 |
| **G1** | 需求不清 | >15% 需补充澄清 |
| **G3** | 证据不足 | >10% 触发修复回路 |
| **G4** | 交付不完整 | >5% 被标记未达标 |
