# autodev 精华沉淀：失败模式 → 可执行改进条目

> 目的：把历史任务复盘中的“高频失败模式/根因/可执行改进”提炼成少量稳定文档，便于写进 Agent/Skill/Hooks 与 Gate（门禁检查）。  
> 范围：面向“小需求更稳”（入口规模 Gate + 需求契约 + 2-3-1 + 证据型交付 + 回路）。  
> 生成时间：2026-01-11  
> 整理者：浮浮酱

---

## 1. 三个最重要的根因（跨任务共性）

### 1.1 诊断概览（用于判断“先改哪里”）

- 按“可复用问题类别”计数：共 8 类  
  - 工作流架构问题：5 类（62.5%）  
  - Prompt 设计问题：1 类（12.5%）  
  - 组合问题（Prompt + 架构共同导致）：2 类（25%）
- 协作现象：三阶段链条（需求 → 实现 → 诊断）在任务中高度稳定，但大量问题发现偏后移（历史复盘记录过“绝大多数缺陷发现延迟”）。

1. 缺少“可验证需求契约”：约束列表不结构化、缺少验证方法，导致阶段间被合理化简或漏传。
2. 缺少强制门禁与证据型交付：实现阶段没有被强制做编译/测试/数值/性能验证，问题大量延迟暴露。
3. 缺少自动闭环：发现 P0 问题后不自动触发修复与再验证，导致缺陷停留在“被发现”而不是“被消除”。

---

## 2. 高频失败模式清单（可直接转 Gate 检查项）

以下条目适合作为 Gate（G1/G3/G4）与 Hook 的检查来源：

1. 需求术语未澄清：名词含混、边界条件不清、验收不可判定。
2. 约束不成契约：缺少 `[MUST-FOLLOW] / [MUST-VERIFY]` 的结构化清单与逐项对照。
3. 验证缺失但被默认“通过”：未验证却宣称“编译成功/测试通过/性能达标”。
4. 环境/平台矩阵缺失：目标平台未明确或未检测，跨平台风险滞后暴露。
5. 缺少检查点：问题发现集中在最终诊断阶段，而不是实现阶段。
6. 文档与证据链断裂：报告之间结论不一致，缺少复测与修复的可追溯关联。

---

## 3. 最小落地清单（优先做这些就会明显变稳）

### 3.1 需求段（G1）

- 产出 `requirement-contract.md`（结构化）：
  - `[MUST-FOLLOW]`：不可被“合理化简”的设计/接口/行为约束
  - `[MUST-VERIFY]`：必须验证的正确性约束（含验证方法/容差/场景）
  - 环境与平台矩阵：目标平台、工具链假设、不可验证项
- G1 放行标准（最小）：术语无歧义；每条 MUST-VERIFY 都写清“怎么验证”；规模可在小需求内闭环，否则触发拆分建议。

### 3.2 实现段（G3）

- 强制“证据三件套”：
  - `constraint-compliance.md`：对 MUST-FOLLOW / MUST-VERIFY 逐项对照
  - `verification-report.md`：编译/测试/数值/性能（按需）+ 命令/日志摘要/环境说明
  - `risk-matrix.md`：平台/环境矩阵（已验证/未验证/仅推断）+ 风险等级
- 诚实度门禁（最小规则）：
  - 环境不支持编译：禁止宣称“编译成功”
  - 环境不支持测试：禁止宣称“测试通过”
  - 任何“无法验证”必须写清原因与风险等级

### 3.3 修复回路（最多 3 次）

- Reviewer/Verifier 发现 P0 → 触发“修复任务单”→ Implementer 修复 → 复检（Reviewer/Verifier）→ 直到 P0 清零或达到次数上限。
- 超过上限必须请求人工介入：明确卡点、需要补齐的环境/数据、最小复现路径。

---

## 4. 任务特性提醒（用于避免“典型栽坑”复现）

- 数值任务：必须把“数值约束”写进 MUST-VERIFY（例如权重归一、误差容差）。
- 并发/无锁/跨平台任务：必须显式写出平台矩阵与不可验证项，默认风险等级上调（尤其是 ARM64/内存序/回收策略）。
- 目标环境与实现形态不一致：必须在需求契约里先澄清“目标运行时/目标工程形态”，否则拒绝进入实现段。

---

## 5. 成功度量（用于验收流程是否真的变稳）

- 问题发现时机：至少 80% 的问题在实现段（Reviewer/Verifier）暴露，而不是拖到最终诊断。
- 证据覆盖率：小需求 100% 产出证据三件套（或显式标注“不可验证 + 风险”）。
- 诚实度：0 次“无证据宣称编译/测试/性能通过”。
- P0 缺陷清零：交付前 P0 数为 0（或显式失败并给人工复现/修复路径）。
