# ChatGPT 长对话精华提炼（2026-03-11）

> 来源：`docs/ChatGPT - AI Coding 编排方法论 .md`。
> 目的：把长对话压缩成可执行的项目决策输入。

---

## 1) 精华内容：核心原则、可用 idea、终极目标

### 1.1 核心原则（最值得保留）

1. **必须有单一主导端（控制面）**
   - 主导端不是最强编码者，而是“任务状态机拥有者”：负责目标边界、阶段裁决、收尾/转阶段决策。
2. **执行与白盒可并，黑盒尽量独立**
   - 白盒更贴实现；黑盒必须从需求与用户视角验收，避免实现偏见。
3. **“代码完成”不等于“任务完成”**
   - 完成定义必须包含：变更记录、决策记录、验证证据、风险与后续事项。
4. **Prompt/上下文必须定期清场**
   - 沉淀长期规则与事实，删除阶段性推理噪音，控制上下文熵增。
5. **闭环优于堆叠 agent**
   - 更重要的是“目标→执行→验证→沉淀→清场”的完整反馈回路，而不是 agent 数量。

### 1.2 有用 idea（可落地）

1. **把流程显式建模成控制回路**
   - Reference(目标) → Controller(主导端) → Actuator(执行) → Sensor(白/黑盒) → Feedback(裁决) → Memory(文档) → Reset(清场)。
2. **引入“目标列表持续更新”机制**
   - 在任务推进中强制更新中间目标清单（完成/新增/降级优先级），避免人和 AI 都忘记维护 task list。
3. **定义“任务前讨论基线 + 提交前更新基线”**
   - 任务开始前统一需求契约；commit 前强制更新 AI-friendly 文档库。
4. **把“方法论”转为“流程强制”**
   - 从“靠自觉遵守”升级为“门禁 + 模板 + hook 的被动约束系统”。
5. **先做稀疏、结构化文档图，再考虑检索增强**
   - 文档结构先稳定，再引入 BM25/RAG；否则检索只会放大噪音。

### 1.3 你的终极目标（从对话反推）

可归纳为一句话：

> **把 AI Coding 从“靠对话技巧的临时协作”升级为“可长期维护、可验证、可复盘的闭环控制系统”。**

展开为三个子目标：

1. **控制感**：你始终掌握目标函数、阶段门禁和最终裁决权。
2. **稳定性**：任务不因上下文漂移而失控，质量问题能在流程内尽早暴露。
3. **长期记忆**：每次交付都反哺文档库，让下一次任务更快、更准。

## 2) 实现终极目标的技术路径

> 结论先行：**不是二选一**。推荐“Skill/Prompt 编排先行，检索能力后置增量”。

### 路线 A：仅靠 Vibe Prompt/Skills 设计（短期可行，ROI 最高）

适用阶段：当前项目就可以启动。

最小实现：

1. 在 `autodev` 流程里增加固定阶段：
   - 目标建模 → 执行 → 白盒 → 黑盒 → 文档更新 → 清场。
2. 增加两类强制产物：
   - `goal-list.md`（目标状态台账）
   - `knowledge-update.md`（本次 commit 的知识增量）
3. 用 hooks 做“诚实度门禁”：
   - 无证据不得声称测试通过；未更新知识库不得进入交付。

优点：改造快、与现有 aw-kernel 一致。  
缺点：跨任务记忆与检索仍依赖人工导航。

### 路线 B：Prompt/Skills + 轻量检索（推荐中期）

适用阶段：A 稳定后。

最小实现：

1. 构建结构化文档索引（目录 + 标签 + 段落锚点）。
2. 先上 **BM25**（本地全文检索）用于“找得到”，不急于向量化。
3. 在 Skill 中增加“检索前置步骤”：
   - 接到任务先检索历史约束、同类决策、已知风险，再生成计划。

优点：实现简单、可解释性强、调试成本低。  
缺点：语义泛化能力弱于向量检索。

### 路线 C：Prompt/Skills + BM25 + RAG（长期增强）

适用阶段：文档规模和复用密度明显上升后。

最小实现：

1. 双通道检索：BM25（精确召回） + 向量（语义召回）。
2. 加 rerank 与证据引用，避免“看起来相关但不可用”。
3. 把检索结果绑定到阶段门禁（计划/验证/交付都要引用证据）。

优点：跨项目/跨时间复用能力强。  
缺点：系统复杂度、维护成本、评估成本显著增加。

### 选择建议

- **现在**：A（立刻做）
- **1-2 个迭代后**：A+B
- **出现明显检索瓶颈后**：A+B+C

## 3) 对当前项目架构需要多大变化？

### 3.1 现状判断

当前仓库已具备：

- Agent + Skill 架构；
- 小需求 Gate 与证据型交付思路；
- 文档沉淀导向。

所以不需要推倒重来，属于**中小幅度演进**而非重构。

### 3.2 架构改造量评估（按阶段）

#### Phase 1（小改，约 1-2 周）

目标：把“终极目标”映射到现有流程，不引入新基础设施。

- 调整 `autodev`：新增“目标列表维护 + 结束清场 + 知识更新”三个硬步骤。
- 新增两个模板：`goal-list.md`、`knowledge-update.md`。
- Hook 增加检查：缺少知识更新或目标列表状态不完整则警告/阻断交付。

**影响面：小（主要是 Skill 文本、模板、hooks 规则）。**

#### Phase 2（中改，约 2-4 周）

目标：增强跨任务复用能力。

- 为 `docs/` 建立统一元数据（标签、日期、主题、状态）。
- 引入本地 BM25 检索命令，接入 Skill 前置步骤。
- 输出“检索证据索引”到交付文档。

**影响面：中（新增检索脚本与索引约定，但不破坏主流程）。**

#### Phase 3（可选中改，约 4-8 周）

目标：在高复杂任务下提升召回质量。

- 引入向量索引/RAG 管线（可与 BM25 并行）。
- 建立检索评估集（命中率、可用率、幻觉率）。
- 将“证据引用”变成交付门禁的一部分。

**影响面：中到大（引入新服务与评估体系）。**

### 3.3 最终建议（针对你这仓库）

1. **先别急着上重型 RAG**：先把流程纪律做硬，收益更确定。
2. **把“目标列表更新”做成第一优先级**：这是你明确提出的核心痛点。
3. **把“提交前文档更新”自动化成门禁**：这一步是长期记忆形成的关键。
4. **把 BM25 作为第一代检索层**：轻、稳、可解释，足够支撑当前规模。
