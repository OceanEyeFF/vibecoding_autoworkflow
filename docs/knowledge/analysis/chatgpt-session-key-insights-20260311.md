# ChatGPT 长对话精华提炼（v2，基于当前项目状态）

> 来源：`docs/ChatGPT - AI Coding 编排方法论 .md`。
> 适用前提：项目已进入“去 Claude 化 + 文档治理”阶段，本提炼以当前路线（Sprint / roadmap 导向）为准。

---

## 0) 本次修订目的

上一版更偏“底层能力路线（Prompt/Skill/BM25/RAG）”。
本次修订聚焦两个现实问题：

1. 在 **已有 MyClaude 参照** 的情况下，哪些 idea 还值得做、哪些可降级？
2. `planning`、`idea` 与 `docs` 的目录关系应如何治理？

---

## 1) 长对话仍然成立的“高价值内核”（应保留）

1. **控制面必须唯一**：主导端负责阶段切换与裁决，而不是“多 agent 自组织”。
2. **验证视角必须分离**：执行/白盒可以贴近；黑盒需从需求侧独立验收。
3. **Done = 代码 + 证据 + 沉淀 + 清场**：否则只是“局部完成”。
4. **目标列表必须持续更新**：不更新中间目标清单是高频失稳点。

这些原则是“方法论资产”，不会因为是否采用 MyClaude 而失效。

---

## 2) 在有 MyClaude 的情况下：idea 的开发必要性判断

> 结论：**有必要，但要重排优先级**。先做“治理层/流程层”，后做“底层引擎层”。

### 2.1 继续做（高必要）

1. **任务前讨论基线**（需求契约、边界、完成条件）
2. **提交前文档更新门禁**（至少更新变更、验证、风险、后续）
3. **目标列表状态化**（完成/新增/降级优先级）
4. **证据化交付纪律**（No Evidence, No Output）

原因：这些是你项目当前差异化能力，且与“文档治理”直接同向。

### 2.2 降级为“按需探索”（中低必要）

1. 一上来做重型 RAG 基建
2. 过早做复杂多 agent 控制器
3. 在文档结构未稳定时做语义检索优化

原因：这些属于“底层工程放大器”，不是当前阶段主矛盾。

### 2.3 与 MyClaude 的推荐分工

- **MyClaude 擅长**：可安装性、模块化工作流分发、通用工程脚手架。
- **本仓库应聚焦**：流程治理（门禁/契约/证据）、项目级知识维护纪律、可审计交付。

一句话：

> 你现在更该做“治理内核”，而不是重复造一套通用工作流平台。

---

## 3) 技术路径（按当前阶段重排）

### Phase A（现在就做）——纯治理增强

- 把 Sprint/roadmap 中的关键约束固化进模板与 gate：
  - `goal-list.md`（目标状态台账）
  - `knowledge-update.md`（提交前知识增量）
- 把“未更新目标列表/未更新知识库”纳入交付阻断或强警告。

### Phase B（下一阶段）——轻检索辅助

- 在文档结构稳定后，上本地全文检索（BM25 或等价能力）
- 仅用于“先找旧约束/旧决策/旧风险”，不改变主流程控制权

### Phase C（可选）——RAG 增强

- 仅在出现明确检索瓶颈时引入向量检索/RAG
- 以“可解释 + 可回溯”作为接入门槛

---

## 4) 回答问题 2：`planning` 和 `idea` 应该与 `docs` 平级吗？

> 建议：**默认不平级**。优先把它们纳入 `docs` 体系，按“生命周期 + 规范性”分层。

推荐结构：

```text
/docs
  /overview          # 路线图、项目状态、读者入口（稳定）
  /planning          # Sprint、迭代计划、进行中任务（中频更新）
  /ideas             # 备选想法、实验提案、待验证假设（高频变化）
  /analysis          # 复盘、提炼、对比研究（沉淀）
```

### 为什么不建议平级（`/planning`、`/idea` 与 `/docs` 同层）

1. **路由复杂度上升**：入口分散，违反“文档单入口”治理目标。
2. **权限/规范分裂**：不同目录容易形成不同写作规范与命名风格。
3. **检索与归档成本更高**：后续做索引、清理、版本化更难。

### 什么时候可以平级

仅在这两类情况：

1. `docs/planning/` 需要被自动化程序当“运行输入目录”（非纯文档）。
2. `idea/` 包含大量原型代码或实验脚本，而非文档提案。

否则默认收敛到 `docs/` 下最稳。

---

## 5) 对当前仓库的最小落地建议（不做大重构）

1. 新增（或确认）`docs/overview/roadmap.md` 作为对外总入口。
2. 将 Sprint 文档保留在 `docs/planning/`，与路线图和任务台账一起维护。
3. 将历史 `IDEA-*` 逐步迁入 `docs/ideas/`（保留 archived 子目录）。
4. 保留现有 `docs/knowledge/analysis/` 作为“复盘与精华沉淀”区。

结论：

> 当前架构应做的是“目录与流程治理收敛”，不是再开一轮底层平台化开发。
