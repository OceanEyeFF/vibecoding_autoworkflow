# AW-Kernel 路线图

> 创建时间：2026-01-06
> 当前基线：v0.5
> 最后更新：2026-03-17

本文件只回答三件事：项目现在优先解决什么、为什么优先、做到什么程度算阶段完成。任务状态、排期、完成记录不在这里维护。

## 何时读我

- 需要了解当前 1-3 个迭代的主线方向
- 需要判断某个提议是否应该进入正式计划
- 需要给 `docs/planning/` 中的任务分配优先级

## 何时不要读我

- 需要看当前谁在做什么：读 `docs/planning/WORKBOARD.md`
- 需要看本周承诺与风险：读 `docs/planning/SPRINT.md`
- 需要看已完成的实际变更：读 `docs/planning/CHANGELOG.md`
- 需要看历史分析背景：读 `docs/knowledge/analysis/`

## 与规划栈的关系

| 文档 | 角色 | 更新频率 |
|------|------|----------|
| `docs/overview/guide.md` | 协作规则与文档路由 | 低频 |
| `docs/overview/roadmap.md` | 战略优先级与退出条件 | 低频 |
| `docs/planning/WORKBOARD.md` | 单一任务台账 | 高频 |
| `docs/planning/SPRINT.md` | 当前迭代承诺 | 周级 |
| `docs/planning/CHANGELOG.md` | 已完成事项记录 | 持续 |

## 当前阶段

**阶段名称**：文档治理收口 + P0 工作流准备  
**时间基线**：2026-03-17

当前判断：

- 规划文档需要先收口到单一状态机，避免 `roadmap / backlog / sprint` 各说各话。
- 1 月份的实验性叙事仍保留在不少活动文档里，只能作为历史背景，不能继续充当当前 SoT。
- 在文档治理完成前，不适合直接推进 `Gate / Contract / Evidence` 的实现细节，否则会继续把规划和实现绑死。

## 优先级总览

### P0：先把交付控制链路定义清楚

| ID | 主题 | 当前状态 | 阶段完成条件 |
|----|------|----------|--------------|
| `P0-0` | 计划文档架构收口 | done | `guide / roadmap / docs/planning` 口径统一，旧 SoT 引用退出主线 |
| `P0-1` | 入口规模 Gate | ready | 有统一的规模判定规则、拆分输出格式和放行条件 |
| `P0-2` | 需求契约模板 | ready | 有可复用模板，明确 MUST-FOLLOW / MUST-VERIFY / 环境矩阵 |
| `P0-3` | 需求门禁 | queued | 需求契约存在可检查的放行清单，未通过不能进入实现 |
| `P0-4` | 证据门禁 | queued | 验证结果必须以证据产物表达，无法验证要显式降级 |
| `P0-5` | Level 0 修复回路 | queued | 失败触发、重试上限、人工介入条件写清并可复核 |
| `P0-6` | 交付整合 | queued | 交付摘要、证据索引、风险/回退格式统一 |

| `P0-8` | 知识沉淀门禁 | queued | 每次交付前都要说明新增知识、更新位置和缺失项 |
