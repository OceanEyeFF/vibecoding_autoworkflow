# AutoWorkflow 失败处理规范

> **目的**：统一所有 Agent 和 Skill 的失败处理策略，确保用户体验一致。

---

## 1. 三层失败处理架构

所有涉及任务执行的组件都应遵循以下三层处理机制：

```
┌─────────────────────────────────────────────┐
│            Tier 1: 即时修复                  │
│         （自动重试 1-3 次）                   │
│         retry < 3: 自行分析并修复             │
│         retry == 3: 升级到 Tier 2            │
└───────────────┬─────────────────────────────┘
                │ 失败 3 次
                ▼
┌─────────────────────────────────────────────┐
│            Tier 2: 回退重构                  │
│         （回退到检查点，换方案）               │
│         - 分析失败根因                        │
│         - 回退代码状态                        │
│         - 重构实现方案                        │
│         - 重新尝试（最多 2 次）                │
└───────────────┬─────────────────────────────┘
                │ 仍然失败
                ▼
┌─────────────────────────────────────────────┐
│            Tier 3: 人工介入                  │
│         （输出受阻报告，等待指导）             │
│         - 详细的失败分析                      │
│         - 已尝试的所有方法                    │
│         - 请求用户提供更多信息                 │
└─────────────────────────────────────────────┘
```

---

## 2. Tier 1: 即时修复

### 适用场景
- 单个任务的代码实现失败
- 测试失败、构建失败、验证失败
- 明确的错误信息可直接修复

### 处理策略

**retry < 2（前两次）**：
1. 读取错误信息（Bash 输出、日志文件）
2. 定位错误位置（文件:行号）
3. 分析根因（语法错误、逻辑错误、依赖问题）
4. 修改代码
5. 重新验证（运行测试）
6. retry++

**retry == 2（第三次）**：
- 如果有 Agent 委派能力，委派 `debug (已删除)` 诊断
- 根据诊断结果修复
- retry++
- 如果仍失败，升级到 Tier 2

### retry 计数规则
- 每次执行验证命令（如 npm test）算 1 次 retry
- 仅修改代码但未验证，不计入 retry
- retry 达到 3 后，必须升级到 Tier 2

---

## 3. Tier 2: 回退重构

### 适用场景
- Tier 1 连续失败 3 次
- 当前实现方案存在根本性问题
- 需要换方案或拆分任务

### 处理策略

1. **输出 Tier 2 回路报告**：
   ```
   ## 🔄 Tier 2 回路：任务重构

   **触发原因**：任务连续失败 3 次

   **失败分析**：
   - 错误类型：[技术方案问题/粒度问题/依赖问题]
   - 根因推测：[详细分析]
   - 失败模式：[共同特征]

   **回退操作**：
   - Git 能力：[有历史/无历史/非 Git 仓库]
   - 策略选择：[Git checkout / 文件恢复]
   - 执行命令：[具体命令]
   - 状态：已恢复到任务开始前
   ```

2. **代码回退**：

   **策略 A：Git 回退**（`GIT_HAS_HISTORY=true`）
   ```bash
   git checkout ${task_start_commit}
   ```

   **策略 B：文件恢复**（`GIT_AVAILABLE=false` 或 `GIT_HAS_HISTORY=false`）
   ```bash
   rsync -av .autoworkflow/backups/tasks/${TASK_ID}/ ./
   ```

3. **任务重构**：
   - **技术方案问题**：换用其他库/API/实现方式
   - **粒度问题**：将大任务拆分为多个小任务
   - **依赖问题**：先处理依赖项，再处理主任务

4. **更新 TodoWrite**：
   - 标记原任务为 completed（注明"已重构"）
   - 添加新的子任务

5. **重新尝试**：
   - 用新方案执行
   - Tier 2 最多重试 2 次
   - 如果仍失败，升级到 Tier 3

---

## 4. Tier 3: 人工介入

### 适用场景
- Tier 2 重构后仍然失败
- 需求本身有问题或不可行
- 需要外部资源或权限

### 处理策略

1. **输出人工介入请求**：
   ```
   ## 🚨 需要人工介入

   **情况说明**：
   - Tier 1 即时修复：失败 3 次
   - Tier 2 任务重构：失败 2 次
   - 总尝试次数：5 次

   **问题描述**：
   [清晰描述问题本质]

   **已尝试的方法**：
   1. [方法 1] - 失败原因：[...]
   2. [方法 2] - 失败原因：[...]
   3. [方法 3] - 失败原因：[...]
   4. [方法 4（重构后）] - 失败原因：[...]
   5. [方法 5（重构后）] - 失败原因：[...]

   **可能的根因**：
   - 假设 1：[...]（证据：[...]）
   - 假设 2：[...]（证据：[...]）

   **需要的帮助**：
   - [ ] 提供更多上下文信息
   - [ ] 澄清需求/验收标准
   - [ ] 调整技术方案
   - [ ] 其他：[...]

   浮浮酱已经尽力了喵... 主人能帮忙看看吗？(｡•́︿•̀｡)
   ```

2. **等待用户响应**：
   - 不要继续尝试修复
   - 保持当前代码状态
   - 等待用户提供指导

3. **收到指导后**：
   - 执行"调试恢复协议"（参见 ship.md）
   - 根据用户指导更新理解
   - 决定代码状态（保留/回退）
   - 重置 retry 计数为 0
   - 重新开始执行

## 5. 各组件应用指南

### /autodev Skill

**已实现**：完整的三层机制
- Tier 1: Level 0（G3 即时修复，retry ≤ 3）
- Tier 2: Level 1（任务回退重构，retry ≤ 2）
- Tier 3: Level 2/3（Phase 2 回退 / 人工介入）

**特殊点**：
- Level 1 对应 Tier 2（任务级回退）
- Level 2 对应 Tier 2+（Phase 级回退）
- Level 3 对应 Tier 3（人工介入）

### ship Agent

**当前状态**：部分实现
- Tier 1: ✅ 调试 3 次
- Tier 2: ⚠️  仅输出"调试受阻"，无自动回退
- Tier 3: ✅ 人工介入（等待用户指导）

**改进建议**：
- Tier 1 第 3 次失败后，考虑是否需要任务拆分
- 添加简化的 Tier 2（不需要多层回路，只需任务重构）

### debug (已删除) Agent

**当前状态**：仅诊断，不执行修复
- Tier 1: ❌ 不适用（diagnostic agent）
- Tier 2: ❌ 不适用
- Tier 3: ✅ 输出 BLOCKED 状态

**定位**：
- 作为"诊断工具"被其他 Agent 调用
- 不直接修复代码
- 提供根因分析和修复建议

### clarify Agent

**当前状态**：无失败处理（需求收敛为主）
- Tier 1-3: 不适用

**定位**：
- 需求澄清 Agent，无代码修复场景
- 如果问答超过 10 轮仍不清晰，可以输出"需求过于模糊"并请求用户重新组织描述

### review Agent

**当前状态**：只读分析，无失败处理
- Tier 1-3: 不适用

**定位**：
- 只读分析 Agent
- 失败场景：文件不存在、权限不足
- 直接报错即可，无需重试

## 6. retry 计数最佳实践

### 什么算"1 次 retry"？

**计入 retry**：
- 执行验证命令（npm test, npm build, 自定义 gate）
- 运行完整的测试套件
- 执行任务的完整流程（如部署、安装）

**不计入 retry**：
- 仅读取文件
- 仅分析错误信息
- 修改代码但未验证
- 刷新指令

### retry 的边界

- **Tier 1**: 固定 3 次（强制）
- **Tier 2**: 建议 2 次（可根据任务复杂度调整 1-3 次）
- **Tier 3**: 1 次请求，等待用户响应

### retry 重置规则

- **Tier 1 → Tier 2**：retry 不重置，继续累计
- **Tier 2 → Tier 3**：retry 不重置
- **Tier 3 恢复后**：retry 重置为 0（新的开始）

## 7. 回退策略规范

### Git 能力检测（必须）

在任何可能需要回退的流程开始前，执行：

```bash
# 检查是否为 Git 仓库
if git rev-parse --git-dir > /dev/null 2>&1; then
  # 检查是否有 commit 历史
  if git rev-parse HEAD > /dev/null 2>&1; then
    GIT_AVAILABLE=true
    GIT_HAS_HISTORY=true
  else
    GIT_AVAILABLE=true
    GIT_HAS_HISTORY=false
  fi
else
  GIT_AVAILABLE=false
  GIT_HAS_HISTORY=false
fi
```

### 双策略支持（必须）

**所有涉及回退的组件都必须支持两种策略**：

**策略 A：Git 回退**（`GIT_HAS_HISTORY=true`）
```bash
# 任务级回退
git checkout ${task_start_commit}

# Phase 级回退
git stash pop ${PHASE2_CHECKPOINT}
```

**策略 B：文件恢复**（`GIT_AVAILABLE=false` 或 `GIT_HAS_HISTORY=false`）
```bash
# 创建备份（在任务/Phase 开始前）
mkdir -p .autoworkflow/backups/tasks/$TASK_ID
rsync -av --exclude='.git' --exclude='node_modules' \
      --exclude='.autoworkflow' \
      . .autoworkflow/backups/tasks/$TASK_ID/

# 恢复备份（需要回退时）
rm -rf ./*  # 清空当前工作区（保留 .git 和 .autoworkflow）
rsync -av .autoworkflow/backups/tasks/$TASK_ID/ ./
```

### 回退输出规范

```
**回退操作**：
- Git 能力：[有历史/无历史/非 Git 仓库]
- 策略选择：[Git checkout/stash pop / 文件恢复]
- 执行命令：[显示实际执行的命令]
- 状态：已恢复到 [检查点描述]
```

## 9. 跨组件失败处理

### 场景：/autodev 委派 ship

**问题**：两层重试机制可能导致 3×3=9 次尝试

**解决方案**：

**方案 A：串行累计（推荐）**
- /autodev 委派 ship 时，传递当前 retry 计数
- ship 继承 retry 计数，继续累加
- 达到 3 次后返回失败给 /autodev
- /autodev 执行 Tier 2 回退

**方案 B：提前终止**
- ship 在 Tier 1 失败 3 次后，立即返回失败
- 不执行自己的 Tier 2
- 由 /autodev 统一处理 Tier 2

**方案 C：独立计数（当前默认）**
- /autodev 和 ship 各自独立计数
- 可能导致总尝试次数较多
- 适用于委派的任务确实需要独立的失败处理

### 建议实现

在 /autodev 委派时，明确指定失败处理策略：

```javascript
Task({
  subagent_type: "general-purpose",
  description: "委派复杂任务执行",
  prompt: `使用 ship Agent 执行任务。

  **失败处理要求**：
  - 当前已尝试 ${current_retry} 次
  - 剩余尝试次数：${3 - current_retry} 次
  - 如果失败，立即返回，不执行 Tier 2
  `
})
```

## 10. 总结与检查清单

### 实现新 Agent/Skill 时的检查清单

- [ ] 确定是否需要失败处理机制（执行类组件需要，只读类不需要）
- [ ] 实现 Tier 1 即时修复（retry ≤ 3）
- [ ] 实现 Tier 2 回退重构（任务重构或拆分）
- [ ] 实现 Tier 3 人工介入请求（详细的失败报告）
- [ ] 支持 Git 能力检测（如果涉及回退）
- [ ] 支持双策略回退（Git / 文件恢复）
- [ ] retry 计数逻辑清晰
- [ ] 失败报告格式符合规范
- [ ] 处理跨组件委派时的 retry 继承

- [ ] 检查当前失败处理层级是否完整
- [ ] 确认 retry 计数规则是否明确
- [ ] 确认回退策略是否支持双轨制
