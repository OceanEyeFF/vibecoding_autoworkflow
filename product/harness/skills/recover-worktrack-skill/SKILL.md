# 恢复工作追踪技能

## 概览

本技能实现 `WorktrackScope.Recover` 状态转移算子，对应 Harness 控制回路中的**恢复控制**阶段。当 gate 判定为失败或阻塞时，它负责评估合法的恢复操作（重试、回滚、拆分、刷新基准、重新规划），选择恢复路径并停在权限边界处。

当 `Harness` 已经判定当前 `工作追踪` 无法沿现有路径继续推进，并且需要一轮限定范围恢复决策时，使用这个技能。

这个技能会为 `通用高能力模型` `SubAgent` 打包最小失败路径上下文，评估合法的恢复操作，在 `重试`、`回滚`、`拆分工作追踪` 或 `刷新基准` 之间做选择，并向 `Harness` 返回结构化的恢复交接结果。

它会停在恢复权限边界处，而不是静默执行破坏性回滚、重定义范围，或改写代码仓库真相。

## 何时使用

当当前问题不是"这个工作追踪是否应该通过关卡"，而是"现在正确的恢复路径是什么"时，使用这个技能：

- 根据限定范围证据归类当前失败或阻塞状态
- 判断当前 `工作追踪` 是否能在相同目标与基准下安全重试
- 判断在任何新尝试之前是否必须先执行回滚
- 判断当前工作是否应拆成更小的限定范围工作追踪
- 判断基准漂移或代码仓库真相变化是否要求 `刷新基准`
- 返回带有明确审批边界和下一步交接结果的恢复决策

## 工作流

1. 确认这是一轮限定范围的 `工作追踪范围.恢复中`，而不是初始规划、实现分派、最终关卡判定或收尾。
2. 载入当前失败路径所需的最小恢复产物。
3. 为限定范围的 `通用高能力模型` `SubAgent` 构建一份 `恢复选择任务简报` 和一份 `恢复选择信息包`。
4. 归类当前恢复触发器：
   - `软失败`
   - `硬失败`
   - `阻塞`
   - `基准漂移`
5. 评估合法的恢复选择：
   - `重试`
   - `回滚`
   - `拆分工作追踪`
   - `刷新基准`
   - `重新规划`（跨 Scope 回到 RepoScope）
6. 选择一条主恢复路径，并在需要时给出一条后备恢复路径。
7. 返回一份固定格式的 `恢复选择结果` 和一份 `恢复权限交接`。
8. 在执行破坏性回滚、开启新工作追踪或变更代码仓库真相之前停止。

- `恢复选项`
- `约束`

## 恢复权限边界

恢复决策必须停留在以下权限限制内：

- `重试`
  - 只有在当前目标、排除目标与基准仍然有效时才允许。
  - 必须检查 `if_interrupted_strategy` 是否允许继续同一 worktrack；如果策略要求 checkpoint、rollback 或 stop，应先转入对应恢复路径。
  - 可以修订下一项限定范围任务或证据要求。
  - 重试时的唯一合法行为是在现有范围、验收定义内重试，并对缺失证据显式暴露。扩大范围、重定义验收或静默跳过缺失证据的行为必须被阻断。
- `回滚`
  - 可以识别回滚目标、回滚原因与安全顺序。
  - 除非程序员明确批准，否则在执行破坏性代码版本或工作区变更前必须停止。
  - 回滚后的唯一合法行为是重新评估底层失败分类。假定仅靠回滚就能解决底层失败分类的行为必须被阻断。
- `拆分工作追踪`
  - 可以提出新的限定范围拆分形态和后续工作追踪身份。
  - 新建工作追踪的创建、调度或切换必须通过显式审批和交接流程。从本技能静默创建、调度或切换到新工作追踪的行为必须被阻断。
  - 必须保留哪些验收标准留在当前工作追踪，哪些会移出。
- `刷新基准`
  - 当上游真相变化或漂移使当前分支比较失效时，可以请求代码仓库级基准刷新或重新初始化触发器。
  - 如果 `baseline_form` 或 `merge_required` 与当前 checkpoint 状态不匹配，必须把该不匹配作为刷新或重新初始化理由。
  - 对 `代码仓库快照/状态`、`目标/章程` 或控制真相的唯一合法操作是读取消费。从本技能改写这些产物的行为必须被阻断。
  - 必须返回一份交接结果，明确告诉 `Harness` 接下来需要哪一轮面向代码仓库的流程。
- `重新规划`
  - 当当前工作追踪的目标、范围或约定本身需要重新审定时，可以触发跨 Scope 回到 RepoScope 的重新规划。
  - 新工作追踪的创建和上游约定的改写必须通过显式审批和交接流程。从本技能静默执行的行为必须被阻断。
  - 必须返回明确的重新规划交接，说明哪些内容需要回到 RepoScope 重新评估。

## 硬约束

- 本技能是控制回路的恢复控制层；负责选择恢复路径并明确权限边界。在本技能中静默执行破坏性回滚、创建新工作追踪或改写代码仓库真相的行为必须被阻断——恢复决策必须停在权限边界处并请求审批。
- 最终 `关卡` 判定结果的唯一合法角色在本技能中是恢复输入。输出最终关卡判定的行为禁止在本技能中发生——那是 `gate-skill` 的职责。
- 仅当恢复决策已完成、审批已通过时，沿恢复路径继续执行才合法。仅因重试看起来可行而继续沿失败路径执行的行为必须被阻断。
- `回滚`、`拆分工作追踪` 或 `刷新基准` 的唯一合法处理方式是作为需要程序员审批的显式恢复决策。将其作为隐藏实现细节的行为禁止发生。
- 当前工作追踪的唯一合法边界是已批准的目标、排除目标与权限边界。超出此边界的行为必须被阻断。
- 当限定范围恢复包已经足够时，传给 `通用高能力模型` `SubAgent` 的唯一合法内容是指定范围恢复包。传入完整代码仓库上下文的行为必须被阻断。
- 变更 `Harness 控制状态`、创建新工作追踪或改写代码仓库真相的行为必须从本技能被阻断。应返回恢复交接结果供监督器决策。
- 恢复分类、恢复选择与审批边界的唯一合法呈现形式是结构化字段。压缩成模糊叙述的行为必须被阻断。

## 预期输出

使用这个技能时，产出一份至少包含以下章节的 `恢复选择结果`：

- `恢复触发器`
- `恢复选项评估`
- `所选恢复路径`
- `恢复权限交接`
- `开放风险与后续跟进`
- `返回 Harness`

结果中至少应包含以下字段或等价表达：

- `子代理模型`
- `恢复触发器`
- `失败分类`
- `当前工作追踪状态`
- `所选恢复模式`
- `选择理由`
- `已应用中断处理策略`
- `节点类型`
- `节点策略`
- `后备恢复模式`
- `重试可行性`
- `回滚目标或规则`
- `拆分理由`
- `基准刷新理由`
- `权限边界`
- `即时安全动作`
- `需要审批`
- `后续产物`
- `开放风险`
- `建议下一动作`
- `需要程序员审批`
- `如何审查`
